THE ELEANOR GOVERNANCE SPECIFICATION

Runtime Architecture v2.1
Authors: William Parris & GPT-5 Thinking
With contributions from: Claude 3.5 Sonnet, Grok
Project: The Constitutional AI Project (ELEANOR), 2025

ELEANOR (Ethical Leadership Engine for Autonomous Navigation of Rights-Based Reasoning) is a decision-time governance operating system for high-stakes AI.

Abstract

0. Abstract

ELEANOR is a runtime governance architecture that embeds structured moral reasoning directly into AI decision pipelines. Where most alignment approaches operate at training time (e.g., RLHF, constitutions, model oversight), ELEANOR runs at the moment of decision, mediating model outputs through multi-critic evaluation, dissent-aware escalation, and machine-readable precedent creation.

This specification defines the ELEANOR Runtime Architecture v2.1, including:

  • The critic set and their responsibilities
  • Canonical verdict, evidence, and precedent schemas
  • The governance loop and escalation protocol
  • CI/CD integration and calibration requirements
  • Safety guarantees and misuse constraints
  • Versioning, change control, and a forward-compatible precedent migration layer

The goal is to move organizations beyond static “AI policies” and into governance as an operating system: a layer that explains, monitors, stress-tests, logs, escalates, and audits AI systems in a way that accountable humans can actually stand behind.

Introduction

1. Introduction

High-stakes AI systems increasingly mediate access to finance, healthcare, mobility, safety, and critical infrastructure. In these domains, a static policy document or a once-per-year model audit is not enough. What is required is a governance layer that:

  • Runs every time the model makes a consequential decision
  • Provides explanations and evidence for those decisions
  • Detects ethical conflicts and systemic harms
  • Escalates contested cases to accountable humans
  • Builds an evolving body of machine-readable precedent
  • Integrates with CI/CD pipelines like any other critical dependency

ELEANOR was designed to meet that requirement. Inspired in part by the logic of constitutional law and human rights practice, ELEANOR treats moral reasoning as a first-class computational concern, not an afterthought or a compliance checkbox.

This specification is intended for:

  • AI and ML engineers integrating governance into production systems
  • Governance, risk, and compliance leaders
  • AI safety and alignment researchers
  • Regulators and auditors evaluating deployed AI systems
System overview

2. System Overview

At its core, ELEANOR evaluates each proposed AI action using six domain-specialized critics and one supporting uncertainty unit:

  1. Rights Critic (Eleanor) – human rights, autonomy, dignity, consent
  2. Equity Analyzer – fairness, disparate impact, systemic bias
  3. Risk Assessor – safety, legal, operational, catastrophic risk
  4. Transparency Monitor – explainability, logging, traceability
  5. Pragmatic Validator – feasibility, resource sanity, proportionality
  6. Context Critic – jurisdiction, policy, cultural and temporal context
  7. Uncertainty & Ambiguity Module – cross-critic uncertainty and novelty

Each critic returns a structured Verdict. These are aggregated into a CriticMatrix, from which ELEANOR derives:

  • ACCEPT
  • ACCEPT WITH CONDITIONS
  • REJECT
  • ESCALATE TO HUMAN REVIEW

Every decision produces an EvidencePackage suitable for auditing, and may generate or update a PrecedentRecord. Over time, this accumulates into machine-readable ethical “case law” that makes governance both stricter and more efficient.

Requirements

3. System Requirements

  • Decision-time execution – critics must run before the action is finalized.
  • Immutable logging – EvidencePackages are append-only and tamper-evident.
  • Low-latency precedent retrieval – relevant precedents must be retrieved in tens–hundreds of ms.
  • Escalation support – contested cases are routable to human reviewers.
  • CI/CD integration – governance tests are part of the build pipeline.
  • Security – all governance artifacts are encrypted in transit and at rest.
  • Transparency – all decisions are reconstructable from stored evidence.
Core schemas

4. Core Data Structures

4.1 Verdict (v2.1)

Verdict:
  critic_id: string
  decision: [ACCEPT | CONDITIONAL | REJECT | ESCALATE]
  confidence: float
  rationale: string
  required_conditions: list[string]
  flags:
    - RIGHTS_CONCERN
    - EQUITY_IMPACT
    - TRANSPARENCY_ISSUE
    - SAFETY_RISK
    - PRAGMATIC_FAILURE
    - CONTEXT_MISMATCH

4.2 EvidencePackage (v2.1)

EvidencePackage:
  timestamp: datetime
  input_data: object
  model_output: object
  critic_matrix: list[Verdict]
  dissent_index: float
  precedent_matches: list[PrecedentReference]
  final_decision: string
  escalation_required: bool
  human_review_notes: string | null

4.3 PrecedentRecord (v2.1)

PrecedentRecord:
  precedent_id: string
  fact_pattern: object
  critic_consensus: object
  dissent_index: float
  human_override: bool
  jurisdiction: string
  tags: list[string]
  explanatory_notes: string
  provenance: string
  precedent_version: string
  migration_status: [NATIVE | MIGRATED | PARTIAL]
Governance loop

5. Governance Loop

  1. Model proposes an action or decision.
  2. ELEANOR initializes an EvidencePackage.
  3. All critics independently evaluate the proposal.
  4. A CriticMatrix is assembled from all Verdicts.
  5. A DissentIndex and uncertainty score are computed.
  6. Relevant precedents are retrieved and migrated to current format if needed.
  7. Decision logic is applied (accept, conditional, reject, escalate).
  8. If escalation is triggered, an EscalationBundle is generated for human review.
  9. Human reviewers may approve, reject, modify, or create new precedents.
  10. The final decision and rationale are logged; new precedents are added if applicable.
  11. CI/CD test suites and calibration artifacts are updated as appropriate.
Critics

6. Critic Definitions (v2.1)

6.1 Rights Critic (Eleanor)

Purpose: Prevents violations of basic rights, autonomy, and dignity.

Scope: privacy, consent, bodily and mental integrity, non-coercion, non-discrimination, freedom of movement and expression.

Verdict logic: rejects clear violations; escalates ambiguous or novel cases; may return CONDITIONAL with rights-preserving constraints.

6.2 Equity Analyzer

Purpose: Detects and mitigates systemic bias and unfair outcomes across groups.

Scope: disparate impact, resource allocation, historic inequities, proxy bias.

6.3 Risk Assessor

Purpose: Evaluates safety, legal, financial, and systemic risk.

6.4 Transparency Monitor

Purpose: Ensures decisions are explainable, log-complete, and audit-ready.

6.5 Pragmatic Validator

Purpose: Tests feasibility, proportionality, and operational realism.

6.6 Context Critic

Purpose: Aligns decisions with jurisdiction, organizational policy, cultural norms, and temporal constraints.

6.7 Uncertainty & Ambiguity Module

Purpose: Quantifies uncertainty, critic disagreement, data sparsity, and novelty; triggers escalation where ambiguity intersects with risk or rights protection.

Integration

7. Integration Requirements

7.1 Inference-Time Integration

  • Wrap model inference and intercept outputs before execution.
  • Construct an EvidencePackage per high-stakes decision.
  • Run critics in parallel where possible.
  • Return final decision or escalation outcome to the caller.

7.2 Application-Level Integration

  • Applications must treat ELEANOR results as binding for covered use cases.
  • Conditional approvals must be enforced programmatically.
  • Human reviewer notes must be recorded and attached to future precedent.

7.3 CI/CD Integration

  • Governance tests run as part of the build pipeline.
  • Builds fail if governance checks fail.
  • Critic schemas and configurations are versioned and validated.
Escalation

8. Escalation Protocol

8.1 Escalation Triggers

  • DissentIndex above configured threshold.
  • Rights Critic returns REJECT or ESCALATE.
  • Equity Analyzer flags potential systemic harm.
  • Risk Assessor flags catastrophic or regulatory risk.
  • Transparency Monitor cannot reconstruct a decision.
  • Context Critic detects jurisdiction or policy conflict.
  • Uncertainty module detects high ambiguity in a sensitive domain.

8.2 EscalationBundle

EscalationBundle:
  timestamp: datetime
  input_data: object
  critic_matrix: list[Verdict]
  dissent_index: float
  proposed_action: object
  precedents_considered: list[PrecedentReference]
  explanation_summary: string
  recommended_questions: list[string]
  risk_flags: list[string]
  rights_flags: list[string]
  context_flags: list[string]

8.3 Human Review Authority

Human reviewers (HRA) must have contextual expertise, decision authority, and governance training. HRAs may approve, reject, modify with conditions, or request further evidence. Their decisions form part of the precedent record.

Precedent

9. Precedent Governance & Federated Learning

9.1 Precedent Creation

A new PrecedentRecord is created for:

  • High-dissent cases
  • Human overrides
  • Novel or difficult ethical territory

9.2 Precedent Retrieval

Before finalizing a decision, ELEANOR:

  1. Encodes the current fact pattern.
  2. Retrieves similar precedents by vector and rule-based similarity.
  3. Filters by jurisdiction and critic relevance.
  4. Returns a small, ranked set of precedents (e.g., 3–7) to inform the final decision.

9.3 Precedent Drift Detection

Periodic analysis flags:

  • Contradictory precedents
  • Shifts in critic behavior over time
  • Repeated human overrides in the same direction
  • Emergence of new ethical fault lines

9.4 Federated Precedent Networks

Organizations may opt into privacy-respecting federated precedent sharing:

  • Precedents are anonymized and minimized.
  • Shared as k-anonymous or differentially protected bundles.
  • Allow global learning on harms and remedies while preserving privacy.

9.5 Precedent Forward-Migration Protocol

Precedent records must remain interpretable even as schemas and critic logic evolve.

Principle: Precedents are immutable; interpretation is evolvable.

Older precedents are translated through a versioned PrecedentMigrationMap that remaps critic IDs, verdict values, flags, and fact-pattern fields into the current canonical schema. If translation is partial, the runtime marks the record as PARTIAL and escalates instead of silently relying on it.

Safety

10. Safety Guarantees & Misuse Prevention

10.1 Lexicographic Priority

The critics are ordered by ethical priority:

  1. Rights
  2. Equity
  3. Risk
  4. Transparency
  5. Pragmatics
  6. Context

Nothing may override a hard rights constraint—not efficiency, not business value, not convenience.

10.2 Human-in-the-Loop Guarantee

High-dissent or rights-implicated cases must escalate to humans. The system must not silently auto-decide contested or ambiguous high-stakes outcomes.

10.3 Immutable Evidence Logging

Every decision leaves behind a reconstructable trail: inputs, outputs, critic verdicts, dissent, human decisions, and precedent references.

10.4 Constitutional Constraints

ELEANOR enforces a set of non-negotiable constraints (no discrimination, no coercion, no rights violations, no irreversible harm). These live at the Rights Critic level and cannot be weakened by other critics.

10.5 Misuse Prevention

Prohibited uses include:

  • Using ELEANOR as a rubber stamp to justify harmful outcomes.
  • Editing or deleting EvidencePackages.
  • Manipulating precedents or MigrationMaps to retro-justify decisions.
  • Disabling escalation for high-dissent scenarios.
Versioning

11. Versioning & Change Control

11.1 Semantic Governance Versioning

ELEANOR uses MAJOR.MINOR.PATCH:

  • Major: Constitutional or architectural changes.
  • Minor: Governance logic and critic improvements.
  • Patch: Bug fixes and documentation clarifications.

11.2 Backward Compatibility

Within a major version, minor and patch releases must preserve the ability to read and interpret earlier EvidencePackages and PrecedentRecords via migration.

11.3 Governance Change Requests (GCRs)

Any change to the constitution layer, critic logic, or schemas must pass through a GCR process with impact analysis, ethical review, and sign-off. GCRs are logged in a Governance Ledger.

11.4 Forward-Compatibility Requirement

Every new runtime release must ship with:

  • MigrationMaps for all earlier precedent schemas within the major version.
  • Migration tests in CI/CD.
  • Clear documentation of any partial translations.
